# E2-micro image pull deploy

Build the app image on GitHub Actions and pull it on the VM. This avoids running dependency-heavy Docker builds on the free-tier host.

The image workflow (`build-app-image.yml`) always publishes an immutable commit-SHA tag (`ghcr.io/kingyx3/poma:<sha>`) with Buildx cache enabled, and moves the mutable `:main` tag only on pushes to `main`. It is a reusable workflow: Auto CI/CD calls it before every deploy so the deploy pulls an image built from the exact ref under test rather than a possibly-stale `:main`. On a pull request it builds the PR head SHA; on a release it builds the released commit. The VM deploy script writes a compose env file, ensures the VM compose file exists, pulls the selected image, runs the smoke test, and installs cron.

Auto CI/CD passes the freshly built SHA image to the deploy workflow via its `image` input, which the deploy job validates and forwards to the VM as `POMA_IMAGE`. When no image is supplied (e.g. a manual `deploy-gcp-vm` dispatch left blank) the deploy script falls back to the `:main` tag. Override the default registry, repository, and tag pieces with DEFAULT_IMAGE_REGISTRY, DEFAULT_IMAGE_REPOSITORY, and DEFAULT_IMAGE_TAG.

The VM-side image pull is bounded at 8 minutes, and the deploy smoke test is bounded at 3 minutes. The deploy script runs timed Compose calls through `timeout_compose`, which invokes `docker compose` directly so GNU `timeout` does not try to execute a shell function named `compose`. Local development can keep using docker-compose.yml with build enabled; the VM uses the deployed docker-compose.vm.yml image reference. The GitHub image build pins `APP_UID=1000` and `APP_GID=1000`, and Terraform startup creates the `poma` app user/group with the same IDs so the pulled container user can write the host runtime directories. The deploy script checks that runtime identity before pulling, so an old VM that has not been recreated by Terraform fails before the smoke test.
